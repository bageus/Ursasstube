# Debug guide (quick)

## 1) Быстрый triage

1. Воспроизведите баг локально (`npm run dev`).
2. Зафиксируйте сценарий (шаги + ожидаемое/фактическое).
3. Прогоните базовые проверки:

```bash
npm run check
npm run build
```

## 2) Gameplay / runtime

- Проверяйте модули в `js/game/` и `js/phaser/`.
- Для lifecycle-проблем (init/destroy/resize) сначала запускайте unit-сеты из `test:request`.
- Если баг проявляется только после нескольких запусков рана, проверьте cleanup и подписки lifecycle.

### FPS сильно просел: быстрый чеклист

1. Замерьте baseline FPS в DevTools Performance (минимум 30–60 секунд) и зафиксируйте:
   - средний FPS;
   - длинные кадры больше 16.6ms;
   - main-thread long tasks.
2. Проверьте, не растёт ли количество объектов после рестарта/смены сцены:
   - активные таймеры/интервалы;
   - подписки на события;
   - частицы/спрайты, которые не уничтожаются.
3. Сузьте причину отключением тяжёлых подсистем по одной:
   - post-processing/blur/glow;
   - частицы;
   - частые `setState`/перерисовки UI;
   - realtime-анимации в фоне.
4. Проверьте графические ассеты:
   - слишком большие текстуры (больше 2048) и их количество;
   - отсутствие atlas/spritesheet для часто используемых иконок;
   - декодирование изображений в runtime (вместо preload).
5. Проверьте частоту обновлений логики:
   - уменьшите частоту не-критичных задач (метрики, polling, визуальные эффекты);
   - объедините мелкие батчи обновлений в один тик.
6. После фикса снова снимите Performance trace и приложите сравнение до/после.


### FPS сильно просел: что приложить в задачу

Минимальный набор артефактов для тикета:

- Chrome Performance trace (`.json`) на 30–60 секунд.
- Скрин вкладки Memory (до и после 2–3 рестартов сцены).
- Версия браузера/OS/GPU и разрешение экрана.
- Коммит/ветка и включённые флаги качества графики.

Быстрые команды для локальной проверки перед фиксом:

```bash
npm run dev
npm run check
npm run test:request
```

Если просадка коррелирует с сетью (рывки раз в N секунд), дополнительно проверьте:

- частоту polling в клиентских сервисах (`js/request.js`, `js/analytics-delivery.js`);
- burst-ответы API, которые приводят к массовым UI-обновлениям за один кадр.

## 3) Store / economy

- Точки входа: `js/store/`.
- Проверяйте, что после покупки/использования ride состояние и UI синхронизированы.
- Для регрессий экономики прогоняйте тесты math/analytics/store контрактов.

## 4) Network/API

- Сетевой контракт централизован в `js/request.js`.
- Если есть падения на JSON/HTTP, проверяйте использование `requestJson` / `requestJsonResult`.
- Для нестабильности запросов проверяйте профиль timeout/retries соответствующего endpoint.

## 5) Analytics

- Эмиссия событий: `js/analytics.js` и доменные сервисы.
- Доставка/батчинг: `js/analytics-delivery.js`.
- Отчётность: `js/analytics-metrics.js` + `scripts/build-product-metrics-report.mjs`.

## 6) Перед отправкой PR

Минимум:

```bash
npm run test:request
npm run test:e2e-smoke
npm run check
```
