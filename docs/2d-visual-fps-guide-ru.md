# Как улучшить визуал 2D браузерной игры без просадок FPS

## Ключевая идея
Текущий подход «FPS упал → резко понижаем качество» действительно часто выглядит плохо: заметные скачки резкости, отключение эффектов и «плавающая» картинка.

Лучше перейти к **бюджетной модели рендера**:
- заранее фиксировать бюджет кадра (например, 16.6 ms для 60 FPS),
- распределять его между системами (симуляция, рендер, VFX),
- адаптировать нагрузку **мягко и локально**, а не глобальным обрушением качества.

---

## 1) Сначала измерение, потом оптимизация

### Что обязательно мерить
- CPU кадр: update, collision, AI, scripting.
- GPU кадр: draw calls, overdraw, fill-rate, shader time.
- Пики (p95/p99), а не только средний FPS.
- Отдельно для слабых устройств (мобильные + старые ноутбуки).

### Практика
- Ввести внутренний профайлер с разбиением по подсистемам.
- Логировать «кто съел кадр» при превышении 16.6/33.3 ms.
- Делать сцену-бенчмарк (стресс-тест эффектов, частиц, света).

---

## 2) Архитектура обновления: стабильная симуляция + рендер с интерполяцией

Используйте паттерн:
- фиксированный шаг симуляции (`fixed timestep`, например 30/60 Hz),
- рендер отдельно через `requestAnimationFrame`,
- интерполяция состояния между шагами.

Это дает:
- предсказуемую физику/геймплей,
- меньше «дребезга» при микро-лагов,
- возможность деградировать **рендер-часть**, не ломая логику.

---

## 3) Режим фиксированного качества (без авто-ухудшения при просадке FPS)

Если вы не хотите видеть деградацию картинки во время игры, можно полностью отключить автоснижение качества и держать визуал стабильным.

### Что делать вместо динамического ухудшения
1. **Фиксированный VFX-бюджет с самого старта**
   - не переключать качество на лету,
   - держать постоянный лимит частиц и слоев эффектов,
   - проектировать эффекты так, чтобы укладываться в бюджет даже на слабых устройствах.
2. **Оптимизировать симуляцию, а не «резать» картинку**
   - object pooling,
   - culling/offscreen sleep,
   - spatial partition для коллизий.
3. **Дешевле считать пост-эффекты, но всегда одинаково**
   - bloom/blur считать в пониженном буфере (например 1/2),
   - не менять это значение по FPS в runtime.

### Чего лучше избегать
- Любого автоматического переключения high/low при просадках.
- Резкого глобального render scale всей игры.
- Частых переключений качества каждый кадр.

---

## 4) Самые частые узкие места в 2D и как их закрывать

### CPU bottlenecks
- Много мелких объектов в JS (GC-паузы).
- Дорогие коллизии «каждый с каждым».
- Избыточные обновления объектов вне экрана.

Решения:
- object pooling для пуль/частиц/временных сущностей,
- spatial partitioning (uniform grid / quadtree) для коллизий,
- culling + «sleep» для offscreen/далеких сущностей,
- вынести тяжёлые вычисления в Web Worker.

### GPU bottlenecks
- Слишком много draw calls.
- Overdraw (полупрозрачные спрайты поверх друг друга).
- Тяжелые full-screen шейдеры.

Решения:
- batching + texture atlases,
- сортировка рендера для минимизации переключения состояний,
- ограничение слоёв прозрачности,
- downsample для пост-эффектов,
- заранее запечённый свет/тени где возможно.

## 5) Технологии для 2D-эффектов в браузере


## 5.1) Лицензии и стоимость (важно)

- **PixiJS** — бесплатный open-source фреймворк (обычно используется по MIT-лицензии).
- **Phaser 3 (CE/open-source ветка)** — также бесплатный open-source (MIT).
- То есть для большинства инди/коммерческих веб-игр эти движки можно использовать **без покупки лицензии**.

Что может быть платным на практике:
- сторонние плагины, ассеты, редакторы, middleware;
- хостинг/CDN/аналитика/бэкенд-сервисы;
- кастомная разработка и поддержка.

Рекомендация: перед релизом проверить актуальные лицензии в официальных репозиториях/сайтах конкретных версий, которые вы используете.

---

## A) PixiJS + шейдеры (WebGL)
Когда выбирать:
- 2D-игра со спрайтами и насыщенными VFX,
- важны быстрый старт и зрелая экосистема.

Плюсы:
- высокая производительность на 2D,
- удобные фильтры/пост-эффекты,
- хорошие инструменты батчинга.

Минусы:
- сложные пайплайны шейдеров потребуют ручной оптимизации.

## B) Phaser 3 (WebGL/Canvas fallback)
Когда выбирать:
- нужен быстрый production с большим количеством готовых систем.

Плюсы:
- богатый набор «из коробки»,
- быстро собирать gameplay.

Минусы:
- для экстремальной производительности/кастомного рендера иногда меньше гибкости, чем у «низкоуровневого» стека.

## C) Custom WebGL2 (или regl) для собственного 2D рендера
Когда выбирать:
- у вас уже есть опыт графического пайплайна,
- нужен полный контроль над батчингом, шейдерами и памятью.

Плюсы:
- максимум контроля и оптимизации.

Минусы:
- выше стоимость разработки и поддержки.

## D) WebGPU (прогрессивно)
Когда выбирать:
- проект на долгий срок, есть ресурсы на R&D,
- хотите современный API и compute-возможности.

Плюсы:
- высокий потенциал производительности,
- лучше масштабируется для сложных эффектов.

Минусы:
- совместимость и tooling пока не идеальны для всех браузеров/устройств,
- нужен fallback (обычно WebGL2).

## E) WASM для тяжёлой логики
Это дополнение, не рендер-движок.

Подходит для:
- физики, pathfinding, сложной симуляции частиц.

Важно:
- переносить в WASM только «горячие» участки, измеренные профайлером.

---

## 6) Рекомендуемая стратегия миграции (без большого риска)

1. Внедрить профайлинг и бюджеты кадра.
2. Перейти на мягкую адаптацию качества (particle/postFX budgets + hysteresis).
3. Оптимизировать CPU (pooling, spatial partition, culling).
4. Оптимизировать GPU (batching, atlas, overdraw control).
5. Только после этого принимать решение о смене технологии.

Практически:
- если вы уже на Canvas2D и эффектов много — переход на PixiJS/WebGL обычно дает лучший «цена/результат».
- если уже WebGL, но FPS проседает — чаще проблема в архитектуре update/render и бюджете эффектов, а не в самой технологии.

---

## 7) Мини-чеклист «визуал ↑, FPS стабилен»

- Есть фиксированный timestep + интерполяция.
- Есть p95/p99 мониторинг кадра.
- Есть draw-call budget на сцену.
- Частицы и пост-эффекты живут в собственных бюджетах.
- Есть hysteresis для автокачества.
- Нет резкого глобального downscale.
- Offscreen объекты не обновляются в full-rate.
- Тяжелые вычисления вынесены из main thread.

---

## 8) Что выбрать прямо сейчас

Если нужна практичная рекомендация для большинства 2D браузерных игр:
- **Рендер:** PixiJS (WebGL).
- **Эффекты:** кастомные шейдеры + downsampled postFX.
- **Логика:** fixed timestep, object pooling, spatial partition.
- **Адаптация:** только мягкая, бюджетная, с hysteresis.
- **Опционально:** тяжелую симуляцию вынести в Worker/WASM.

Это обычно дает заметно более «дорогую» картинку без скачков качества и с устойчивым FPS на массовом железе.
