# Ursasstube — проверка тонких мест для взлома/читов

Дата: 2026-04-18

## Что было проверено
- Клиентская отправка результатов в лидерборд (`js/api.js`).
- Клиентская покупка апгрейдов (`js/store/upgrades-service.js`).
- Auth-поток (wallet/telegram) на стороне фронтенда (`js/auth-service.js`, `js/auth-state.js`).
- Локальное хранение прогресса/рекордов (`js/state.js`).

---

## Ключевой вывод
Клиент (браузерный JS) по определению недоверенная среда. В текущей архитектуре часть античит-ограничений реализована на клиенте (валидация, подпись, локи UI), но **это можно обойти через DevTools/скрипты**. Поэтому критичная защита должна быть на backend.

---

## Уязвимые и «тонкие» места

### 1) Leaderboard: результат формируется полностью на клиенте
**Где:** `saveResultToLeaderboard()` формирует `score/distance/goldCoins/silverCoins` из клиентского стейта и отправляет в `/api/leaderboard/save`.  
**Риск:** High.

Почему это тонкое место:
- Любой пользователь может в рантайме изменить `gameState` через DevTools перед отправкой.
- Wallet-подпись подтверждает владение ключом, но **не доказывает честность gameplay**.
- Для Telegram-режима подписи EIP-191 нет (другой trust-модель), значит серверу нужен особенно строгий verify и rate-limit.

Рекомендации:
1. На backend ввести проверку правдоподобия run (max score growth, max coin gain, tempo событий).
2. Использовать серверный `run_id` + серверный nonce + short TTL, отклонять повторные submit/replay.
3. Отдельный антифрод-флаг для «подозрительных, но не фатальных» run (карантин таблицы).

---

### 2) Store purchase: клиентский tier/ключ можно подменить
**Где:** `buyUpgrade(key, tier)` отправляет `upgradeKey`, `tier` в `/api/store/buy`.  
**Риск:** High.

Почему это тонкое место:
- UI уже ограничивает порядок покупки, но это только клиентский контроль.
- Через консоль можно отправить произвольный `tier`/`upgradeKey`.
- Telegram-ветка без wallet-подписи требует особенно строгой серверной валидации доступа к аккаунту.

Рекомендации:
1. На backend считать цену/доступность **только из серверного состояния**, игнорируя клиентские ожидания.
2. Жёстко проверять последовательность уровней и ownership.
3. Сделать идемпотентность по `request_id` + защита от race/double-spend.

---

### 3) Telegram auth/linking: доверие к полям клиента
**Где:** `authenticateTelegram(...)` и telegram-данные в purchase/save flows.  
**Риск:** Medium/High (зависит от backend verify).

Почему это тонкое место:
- На фронте нельзя считать `telegramId/username` доверенными.
- Если на backend нет обязательной проверки `telegramInitData` (подпись Telegram + TTL), возможен spoofing.

Рекомендации:
1. Серверно верифицировать `telegramInitData` по официальному алгоритму Telegram.
2. Обязательный TTL/anti-replay для initData.
3. Логировать device/IP fingerprint для аномалий по аккаунтам Telegram.

---

### 4) Локальный прогресс в `localStorage` легко изменяем
**Где:** `bestScore`, `bestDistance` читаются/пишутся в `localStorage`.  
**Риск:** Low (если это только UI-метрики), Medium (если влияет на награды/матчинг).

Почему это тонкое место:
- `localStorage` полностью подконтролен пользователю.
- Любые достижения/награды, завязанные на local-only данные, можно накрутить.

Рекомендации:
1. Всё, что влияет на экономику, хранить и считать на backend.
2. Локальные значения использовать только как кэш/UX.

---

## Приоритетный план фиксов (коротко)
1. **P0:** backend anti-cheat для leaderboard + nonce/TTL/replay-protection.
2. **P0:** backend authoritative store validation + idempotency keys.
3. **P1:** строгая Telegram verify + audit trail.
4. **P2:** вынести экономически значимый прогресс из localStorage на сервер.

---

## Что важно
Даже идеальный фронтенд-античит не решает проблему полностью. Нужна server-authoritative модель для любых данных, которые влияют на рейтинг, валюты, апгрейды и награды.
