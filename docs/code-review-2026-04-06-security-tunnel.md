# Ревью от 2026-04-06: улучшения, безопасность и упрощение сцены трубы

## Контекст
Проведён обзор текущей Phaser-реализации туннеля (`MainScene` + `TunnelRenderer` + `tunnel-draw-pass`) и базовых security/network utility.

---

## Ключевые наблюдения

### 1) `TunnelRenderer` всё ещё перегружен константами и доменной логикой
Файл содержит большой блок рендер-констант и вспомогательных мат-функций рядом с lifecycle-классом рендера.

**Почему это проблема:**
- сложнее управлять визуальными пресетами;
- сложно покрывать тестами математику отдельно от Phaser-контекста;
- повышается цена изменений при балансировке VFX.

**Что сделать:**
- вынести константы в `tunnel-render-config.js`;
- вынести геометрию/цветовые helper-функции в `tunnel-math.js`;
- оставить в `TunnelRenderer` только orchestration (create/applySnapshot/destroy + вызовы pass).

**Приоритет:** P1.

---

### 2) `drawTunnelPass` создаёт много короткоживущих объектов на каждый кадр
Внутри двойного цикла depth × segments формируется много `{x,y}` и overlay-объектов (`gridRingOverlays`, `gridRadialOverlays`, `trackSlatOverlays`, `speedStreakOverlays`).

**Почему это проблема:**
- лишняя нагрузка на GC;
- риски микрофризов на слабых устройствах/webview;
- усложнённый профиль производительности из-за аллокаций в hot path.

**Что сделать:**
- перейти на object-pool для quad/overlay;
- либо частично рисовать "на лету" без накопления массивов (там, где порядок не критичен);
- кешировать вычисления углов `sin/cos` для текущего `segmentCount` на кадр.

**Приоритет:** P1.

---

### 3) Нужна явная декомпозиция «сцены трубы» на pipeline-шаги
Сейчас pass делает сразу всё: базовую геометрию, затем сетку, затем slat, затем speed streaks.

**Что сделать (упрощение сцены трубы):**
- ввести pipeline-структуру:
  1. `buildDepthFrame()` — только геометрия + кэш координат;
  2. `renderBaseLayer()`;
  3. `renderGridLayer()`;
  4. `renderTrackLayer()`;
  5. `renderFxLayer()`.
- добавить lightweight профилирование по шагам (`performance.now()`) в debug-режиме.

**Практический эффект:**
- проще выключать/включать эффекты по quality;
- проще локализовать визуальные баги;
- легче проводить A/B оптимизации.

**Приоритет:** P1.

---

### 4) Сглаживание tube-параметров зависит от FPS
`getSmoothedTube()` использует фиксированные коэффициенты (`0.24`, `0.16`) без учёта `delta`, что делает поведение отличающимся между 30/60/120 FPS.

**Что сделать:**
- перейти на delta-based экспоненциальное сглаживание: `alpha = 1 - exp(-k * dtSeconds)`;
- хранить последний `dt` в рендерере и передавать в smoothing-функцию.

**Приоритет:** P2.

---

### 5) Security: сетевой helper надёжный, но есть пробелы hardening
`request()` корректно обрабатывает timeout/retry/abort, но API "низкоуровневое" и легко допустить ошибку на уровне вызова.

**Риски:**
- потребители могут забыть проверку `response.ok`;
- нет отдельного safe-wrapper для JSON и схемы ошибки;
- отсутствует ограничение допустимых протоколов URL.

**Что сделать:**
- добавить `requestJson(url, opts)` c обязательной проверкой `response.ok` и безопасным parse;
- добавить ограничение на протоколы (`http:`/`https:`);
- ограничить пользовательские `timeoutMs/retries` верхними границами для защиты от misuse.

**Приоритет:** P1.

---

### 6) Security: валидация Telegram handle корректная, но fallback лучше централизовать
`sanitizeTelegramHandle()` уже использует строгую regex-валидацию и fallback.

**Что улучшить:**
- вынести default handle в config-константу, чтобы избежать расхождений между модулями;
- добавить тесты на граничные случаи (длина 4/5/32/33, unicode, пробелы).

**Приоритет:** P3.

---

## Быстрый план внедрения
1. **Итерация 1 (без визуальных рисков):** декомпозиция констант/математики + `requestJson` + тесты security helper.
2. **Итерация 2:** object-pool/снижение аллокаций в `drawTunnelPass`.
3. **Итерация 3:** delta-based smoothing + pipeline-раскладка шагов рендера.
4. **Итерация 4:** tuning quality presets по результатам профилирования.

---

## Что проверить после изменений
- стабильность FPS в synthetic smoke;
- отсутствие визуальной деградации mouth/grid/track эффектов;
- корректность retry/timeout/abort и новой JSON-обёртки;
- отсутствие роста ошибок в auth/store API flow.
